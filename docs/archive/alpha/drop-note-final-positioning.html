北极星 — 一句话定位（三人共识）

发给任何东西一封邮件，它就永远在那里，随时能找回来。
"Drop anything to your inbox. Find it back instantly. No app. No extension. Just email."

PRD #1 的"AI-native memory layer"愿景太大、不好传播；PRD #3 的"fastest way to save"太像广告语。 最终采用 PRD #2 的核心承诺框架：零摩擦存、零摩擦找、存了不会忘。一句话定位聚焦"存得进去 + 找得回来"，AI 是执行工具不是卖点。

北极星指标

指标为什么选它
被重看率
被重新打开 / 总收藏 ≥ 20%		 PRD #2 的建议。注册数只衡量获客；被重看率直接衡量"存了有没有用"——这才是和所有书签/收藏类产品的核心差异化点。如果用户存进来从不回看，产品就失败了，再多注册数也没意义。
辅助指标：找回成功率
搜索后有点击 / 总搜索次数		 PRD #3 的建议。搜索是"找回"的直接行为，成功率低说明搜索体验或 AI 标签质量有问题，可快速定位。
激活率（24h 内首存） 衡量注册 → 真实使用的转化，是 retention 分析的前置指标。
注意：注册数、DAU、MAU 不是北极星指标。它们是渠道指标，留给营销看。drop-note 的胜负手在"存了能不能找回"，不在"有多少人注册过"。

议题一：定位叙事 — Omnivore 替代 vs 邮件零摩擦
Mic — drop-note Director

我必须先纠正一件事：我们在 2026-04-02 已经把定位锁成了 Omnivore 替代（D2），但 PRD #2 明确建议"叙事从 Omnivore 难民改为邮件零摩擦"，理由是 Omnivore 社区 10 个月前就散了，靠这个定位难以持续建品牌。这是一个真实的分歧，需要在今天拍板。

我的立场：两手都要，但主叙事变。日常品牌沟通用"邮件零摩擦"——这是可以持续讲的故事，不依赖一个已关闭的产品。Omnivore 变成一个 SEO 落地页策略，悄悄接住搜索流量，但不出现在主页标题里。

Mathieu — Product & Engineering VP

同意 Mic 的双轨策略。但我要补充一个公司视角：我们的 D2 锁定的是"Omnivore 社区是首发渠道"，不是"Omnivore 是品牌核心"——这两件事要分开看。HN/Reddit/r/selfhosted 上冷启动，Omnivore 关键词是流量入口；但首页的 tagline 应该是能活 5 年的定位，不是"我们是 X 的替代品"。

结论：D2 保留发射渠道策略（Omnivore 社区帖子、SEO 落地页），但主品牌叙事调整为"邮件入口的内容收件箱"。这不是推翻 D2，是补充它。

product-manager

从用户调研角度看，PRD #3 说得最准：真正的竞争对手不是 Omnivore，而是"微信收藏 + 浏览器书签 + 截图囤积"这个习惯。Omnivore 用户是高质量种子用户，但不是长期市场。长期市场是"每天看 Reddit/X/Newsletter 但什么都记不住"的内容消费重度用户。

建议：HN 发帖可以用"I rebuilt Omnivore"作为标题钩子（历史证明这类帖子有流量），但 Landing Page 的核心 CTA 应该是："Send anything by email. Find it back instantly." 不提 Omnivore。

共识：叙事双轨

主品牌叙事 = 邮件零摩擦收件箱（"Send anything. Find it back."）。Omnivore = 冷启动渠道 + SEO 落地页，不是品牌核心。D2 的发射渠道策略保留，主叙事升级。

议题三：目标用户 — 技术圈 vs 普通人
Mic

第一版押技术圈，没有争议。 理由：① HN/r/selfhosted 是我们唯一有信心的冷启动渠道 ② AGPL + Docker self-host 天然过滤出技术圈用户 ③ 技术圈用户会贡献 PR、发 bug report、帮你传播 ④ 普通人不配置 MX 记录，所以 hosted 版本是他们的入口，但获客渠道还是技术圈口碑带动。

Mathieu

补充一个资源约束：Jason 一个人发布，无法同时维护两个用户群的 onboarding 路径。技术圈用户自带调试能力，能容忍更粗糙的初体验。优先服务技术圈 = 降低早期支持成本。

product-manager

同意押技术圈，但要注意：技术圈里的目标用户是"内容消费重度用户"，不是"要部署开源工具的 SRE"。具体画像：独立开发者、PM、研究员、重度 Reddit/X/Substack 用户——他们有信息过载的痛苦，但不一定想维护自己的服务器。所以 hosted 版本对技术圈也有吸引力，不能只做 self-host。

共识：技术圈内容消费重度用户

独立开发者、PM、研究员、重度 Reddit/X/Substack 用户。自然扩展：非技术知识工作者（通过技术圈口碑）。坚决不做：学生/学术/重度 PKM 双链玩家。

议题四：Retention — Weekly Digest 是不是必做
Mic

三份 PRD 都把 Weekly Resurface Digest 列为关键 feature，PRD #1 甚至标了"critical"。我同意，这是 MVP 必须做的，不是 V2。理由很简单：用户平时忙，不会主动打开 dashboard。Digest 是唯一能把"存了就忘"变成"存了能想起来"的机制——这直接驱动北极星指标（被重看率）。

但要做对：Digest 不是把最近 7 条发给你——那只是 email newsletter。真正的 Digest 应该是"你 3 周前存的这条，AI 认为现在和你正在关注的主题相关，推给你"。间隔重现，不是线性推送。

Mathieu

同意 Digest 必做，但警告：间隔重现算法是个兔子洞，MVP 阶段要克制。建议 V1 Digest = 每周一封，随机挑 3-5 条 30 天以前存的内容，带 AI 摘要 + "重新打开"按钮。简单，可发布，能驱动北极星指标。算法优化是 V2 的事。

product-manager

支持。另外，Digest 邮件本身就是最好的留存型营销渠道——每周都有 drop-note 品牌露出，而且是在用户本来就要看的邮件场景里。Digest 的 footer 放一行"用这个地址发给朋友，他也能存东西：drop@dropnote.com"，这是零成本的病毒传播。

共识：Weekly Digest 是 MVP 必做，但 V1 版本保持简单

V1 = 每周一封，随机抽取 30 天前存的 3-5 条 + AI 摘要 + "重看"链接。不做复杂的间隔重现算法（V2）。Digest footer 加 referral 钩子。

议题五：AI 成本 — 每条都摘要 vs 按需摘要
Mic

PRD #2 提出了一个真实的成本风险：如果每条 ingest 都立刻调 AI 摘要，成本随用户量线性增长。我们现在用的是 GPT-4o-mini（D8），已经是最便宜的可用模型，但免费用户囤积 50 条内容的摘要成本需要测算。

我的建议：维持现有"每条立刻摘要"的架构，但加一个单用户日成本上限。超过上限的条目排队延迟处理，不是不处理。这比"按需摘要"的用户体验更好（不用点开才摘要），也不会失控。

Mathieu

支持维持自动摘要，但要做一件事：让 Jose 建一个 AI 成本追踪，每周从 OpenAI 账单里拉一次，看每活跃用户的平均 AI 成本。如果超过 $0.10/用户/月，就考虑智能截断（长文本只摘前 2000 tokens）。如果超过 $0.50/用户/月，就讨论按需摘要。现在不需要过早优化，但要有数据。

product-manager

从用户体验角度，自动摘要是核心价值主张的一部分——用户存进来，AI 已经帮他读过了，这个"已读感"很重要。去掉自动摘要就破坏了产品承诺。成本优化应该在不改变用户体验的前提下做：比如对纯图片附件不摘要、对重复 URL 只摘要一次、对超长 PDF 只摘前 N 段。

共识：自动摘要保留，成本控制靠智能截断而非按需

维持"每条 ingest 自动摘要"架构。加单用户日成本软上限（超限排队，不取消）。加 AI 成本追踪（每周拉 OpenAI 账单）。智能截断策略：纯图片不摘要、重复 URL 缓存、超长 PDF 截 2000 tokens。

预发布 Blocker — 发布前必须清零

共 4 个 blocker。3 个派给 Jose，1 个需要 Jason 确认。每个 blocker 都有 brief 模板，窗口一开立刻派发。

P0 JASON
Domain 确认
SendGrid Inbound Parse 需要真实域名的 MX 记录。Jason 曾在 4 月 11 日提到购买域名，当前状态未知。
需要 Jason 确认：① 域名是否已购买？② 域名是什么？③ DNS 是否指向 SendGrid MX？
P0 JOSE
AI 错误处理
当前 worker 没有 response.ok 检查、没有 timeout、无法区分 OpenAI 4xx（不重试）vs 5xx（重试）。P0 级风险，生产环境会静默失败。
Brief 摘要：在 AI 调用外层包 retryWithBackoff，4xx 立刻标记 item 为 error 状态，5xx 最多重试 3 次，加 30s timeout。
P0 JOSE
Dockerfile HOSTNAME
Next.js 在容器内绑定到 localhost，docker compose up 的 web 服务无法从外部访问。self-host 故事完全破碎。
Brief 摘要：在 Dockerfile CMD 或 next.config.js 中设置 HOSTNAME=0.0.0.0，验证 docker compose up 后 localhost:3000 可从宿主机访问。
P1 MIC+JOSE
Legal Pages
/terms 和 /privacy 需要真实内容。数据控制方 = Jason Gao（D5）。AGPL-3.0 许可证要求隐私政策与自托管承诺对齐（不能说"我们出售你的数据"）。
计划：Mic 起草条款内容（中英双语），Jose 负责路由和页面渲染。不是法律文书，是创始人级别的诚实声明。
