## Process rules — LOCKED after Task E audit (2026-05-27)

Jose 的每个 PR 必须满足以下三条，否则 Mic 拒绝接受：

1. **TDD 硬约束** — 所有新 route / 新 lib function 必须先写 `*.test.ts`，见红后写实现，见绿后 commit。PR 里必须有对应测试文件，否则视为不完整。
2. **End-to-end verification 硬约束** — 交工时必须附证据。API route 要 curl 截图（200 + 正确响应体）。UI 变更要 Playwright 截图或浏览器实际截图。Email 功能要 Resend Activity 截图（邮件 delivered 状态）。声称"lint/typecheck/build green"不等于 verification done。
3. **禁止 try/catch 吞错** — 任何新写的 async 函数不能用 `try { ... } catch (e) { console.error(e) }` 模式把错误吞掉。必须让错误向上抛，由调用方（cron handler、route handler）用 `Promise.allSettled` 正确计数 failed，让监控能看到真实失败率。

**背景：** Task E PR #2 Jose 跳过了 TDD，跳过了 email 送达验证。`lib/email.ts` 的 try/catch 导致 Resend 403 被吞，cron 返回 `processed:1` 是假成功，Jason 没收到任何 digest 邮件——直到手动挖 Resend logs 才发现。这三条规矩是为了防止这类"代码绿、产品坏"的情况。

**典范行为（2026-05-27 Task F）：** Jose 在 Task F 中遇到 Google OAuth only 无法做 headless 自动化时，**主动 block 并 ping Mic 请求 manual verify，而不是伪造假截图**。这是新流程第一次真正落地的证据。结果：Jason 本机 manual verify 成功，截图存档，PR #3 有真实的 browser evidence。这个行为模式要保持：证据必须真实，宁可 block 也不伪造。

**Context:**

**What to build:**

**1. Fix `lib/email.ts` — stop swallowing Resend errors (housekeeping, ~5 lines)**

File: `apps/web/lib/email.ts`

Current pattern (BAD):
```typescript
try {
  await resend.emails.send({ ... })
} catch (err) {
  // Never throw — don't break cron flow
  console.error('[email] Failed to send weekly digest email:', err)
}
```

This hides real failures. Resend 403 errors were silently swallowed in Task E, causing false `processed:N` counts in the cron.

Fix: remove the try/catch from `sendWeeklyDigestEmail` and `sendWelcomeEmail`. Let errors throw. The callers (`/api/cron/digest` and the auth callback) must handle errors:
- `/api/cron/digest`: already uses `Promise.allSettled` — it will correctly count `rejected` results as failed
- Auth callback (welcome email): wrap only there with a non-fatal catch, logging the error but not blocking sign-in

After this fix, if Resend is misconfigured or returns a 4xx, the cron's `failed` count will be non-zero and visible in logs.

Write a test: `apps/web/lib/__tests__/email.test.ts` — mock Resend, verify that when Resend throws, `sendWeeklyDigestEmail` propagates the error (does not swallow it).
