Checking Results After Each Send

For each test email, do all three of the following checks before marking Pass.

1
Dashboard check — open https://dropnote.me/items within 60 seconds. The new item should appear at the top of the list. Verify:
  • Item title matches your email subject
  • AI summary is non-empty (not just "Processing..." or blank)
  • At least one tag has been assigned
  • No red error badge on the item card
2
Vercel log check — in your tailing terminal, look for a line matching: 
{"event":"ingest.completed","userId":"...","itemId":"...","status":"done","total_handling_ms":...}
Key fields to verify:
statusMust be "done", not "error" or "failed"
total_handling_msShould be under 30,000 (30s). Alert Mic if over 20,000 consistently.
ai_processing_msTypically 1,000–8,000ms for normal emails. Over 15,000ms = worth noting.
attachment_countFor G3 (PDF test), must be 1, not 0.
3
Supabase item row check (optional, only if dashboard shows nothing) — go to your Supabase project → Table Editor → items table. Filter by user_id = your user UUID. Look for the row. Check:
  • status — should be done
  • error_message — should be null
  • summary — should be non-empty
  • deleted_at — should be null

What To Do If a Test Fails

Work through this triage order. Stop when you find the cause.

Symptom Likely Cause Action
Item never appears in dashboard (not even after 2 min) SendGrid did not deliver to /api/ingest — MX misconfiguration, or from-address not recognized Check Vercel logs: vercel logs --prod https://dropnote.me | grep ingest. If no log line at all, SendGrid never called the webhook. Check SendGrid Activity Feed for the inbound parse event.
Item appears but summary is empty or says "error" OpenAI call failed, timed out, or returned non-OK Check Vercel log line for this item: look for status:"error" or status:"failed". Check error_message on the Supabase row. If OpenAI 429 (rate limit) or 5xx, the retry logic should have caught it — if not, escalate to Mic.
Item appears but no tags Tag extraction returned empty; can happen with very short or ambiguous content Not a hard failure. Check the Vercel log for ai_processing_ms — if it ran successfully and just returned no tags, that is expected for very short emails. Note it for future AI prompt tuning.
Vercel log shows status:"error" Ingest route threw an unhandled exception Copy the full log line and the itemId. Check Supabase items row for error_message. Send the log line to Mic to brief Jose on a fix.
PDF attachment not processed (attachment_count: 0 in log) Attachment MIME type rejected, or attachment size over free tier 10MB limit Verify the PDF is under 10MB. Check Vercel log for any attachment rejected message. If under 10MB and still ignored, send the log to Mic — likely a MIME-type matching bug.
Outlook HTML comes through garbled Outlook's VML / conditional comments leaking into the text extraction Screenshot the item body in the dashboard. Note the garbled HTML. This is a known Outlook quirk and will need a post-QA fix in the HTML sanitizer — escalate to Mic after completing the full matrix.
ProtonMail forwarded email (P3) drops silently Proton rewrites the from-address on forward — the sender no longer matches public.users.email This is a known limitation of the V1 shared-inbox model. Note it. Not a blocker for launch (V2 per-user routing with drop_token is the fix — see docs/v2-per-user-routing.md).
total_handling_ms > 30,000 on any item OpenAI response slow, or attachment processing stalling Note the item ID and total_handling_ms. If it happens on 3+ consecutive emails, escalate to Mic immediately — D11 revisit trigger may have been hit (p95 approaching 30s threshold).
